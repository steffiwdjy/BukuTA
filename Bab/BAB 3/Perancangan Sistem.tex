\section{Perancangan Sistem}

Berdasarkan hasil analisis sistem dan wawancara dengan pihak terkait, sistem dirancang untuk mendigitalkan proses pencatatan yang sebelumnya manual dan belum terintegrasi, sekaligus untuk mendeteksi penyewa yang berpotensi melakukan pelanggaran. Pendekatan ini berfokus pada integritas data, keamanan akses, dan kemudahan pemantauan melalui notifikasi jika data yang diinput merupakan penyewa yang berpotensi melakukan pelanggaran.

Informasi yang diperoleh melalui observasi lapangan, studi dokumentasi, serta wawancara dengan pengelola apartemen menjadi masukan penting dalam penyusunan arsitektur sistem. Misalnya, saat Pelaku Komersil hendak mendaftar harus input NIB (Nomor Induk Berusaha) agar jika suatu saat Pelaku Komersil melakukan pelanggaran akan lebih mudah untuk diproses hukum. Selain itu, sistem perlu memperhatikan bahwa pola penyewaan jangka pendek seperti harian atau mingguan dapat berbeda secara perilaku dari penyewaan bulanan. Oleh karena itu, sistem dikembangkan dengan mempertimbangkan fleksibilitas input jenis sewa. Selain pencatatan data utama, sistem juga mencatat seluruh aktivitas perubahan data yang dilakukan oleh Pelaku Komersil dalam bentuk \textit{log history}, sebagai jejak digital yang bisa digunakan untuk audit atau konfirmasi jika terjadi ketidaksesuaian data.

\subsection{Arsitektur Sistem}
Sejalan dengan landasan teori mengenai teknologi web yang telah diuraikan pada Subbab~\ref{sec:node_js}, sistem {data capture} ini dibangun menggunakan arsitektur \textit{client-server}. Spesifikasi teknis dari komponen arsitektur tersebut disesuaikan dengan kebutuhan integrasi dan efisiensi sistem sebagai berikut.

\begin{itemize}
    \item \textbf{Antarmuka Pengguna (\textit{Frontend}):} Dibangun menggunakan pustaka \textbf{React.js} dan perkakas \textit{build} \textbf{Vite} untuk menghasilkan antarmuka web interaktif (\textit{Single Page Application}). Pertukaran data dengan server ditangani oleh \textbf{Axios}.
    \item \textbf{Sistem \textit{Backend}:} Sebagaimana teknologi yang dipelajari pada Subbab~\ref{sec:node_js}, bagian ini dibangun menggunakan \textbf{Node.js} dengan kerangka kerja \textbf{Express.js} untuk memproses logika bisnis dan menyediakan layanan \textit{RESTful API} secara asinkronus.
    \item \textbf{Basis Data:} Menggunakan \textbf{MySQL} yang terpusat pada ekosistem SRusun. Akses basis data dioptimalkan dengan pustaka \texttt{mysql2/promise} melalui \textit{connection pooling} agar tangguh menangani koneksi konkuren.
    \item \textbf{Manajemen Berkas:} Pustaka \textbf{Multer} digunakan untuk memproses pengunggahan (\textit{upload}) berkas \textit{multipart/form-data}, memfasilitasi penyimpanan dokumen bukti pelanggaran secara aman.
    \item \textbf{Keamanan dan Autentikasi:} 
    \begin{itemize}
        \item Mengadopsi \textit{Single Sign-On} (SSO) terpusat, mengeliminasi kebutuhan penyimpanan kata sandi lokal.
        \item Otorisasi hak akses menggunakan \textbf{JSON Web Token (JWT)} dari ekosistem SRusun melalui modul \texttt{jsonwebtoken}.
        \item Keamanan interaksi lintas domain dijaga menggunakan modul \textit{middleware} \textbf{CORS} (\textit{Cross-Origin Resource Sharing}).
    \end{itemize} 
    \item \textbf{Ekspor Data:} Fitur konversi data ke format \textbf{CSV} didukung oleh pustaka \textbf{json2csv} untuk kebutuhan audit dan pelaporan eksternal.
\end{itemize}



\subsubsection{Strategi Deployment} 
Peluncuran aplikasi ke lingkungan produksi (\textit{production environment}) dilakukan secara bertahap dalam tiga fase: Fase Sebelum Integrasi (Modul Mandiri), Tahap Sementara, dan Fase Setelah Integrasi. Perubahan skema ini didasari oleh arahan dan evaluasi dari dosen pembimbing guna memastikan sistem dapat diuji dan digunakan secara optimal. Pembangunan Tahap Sementara bertujuan untuk mempercepat proses pengumpulan data di lapangan, dikarenakan integrasi penuh dengan aplikasi \textit{Member} The Jarrdin memerlukan masa sosialisasi internal terlebih dahulu dengan pihak p3srs.thejarrdin.com.

\textbf{Fase 1: Sebelum Integrasi (Modul Mandiri)}\\
Pada tahap awal pengembangan atas arahan dari dosen pembimbing sistem diunggah (\textit{deploy}) ke server SRusun dengan alamat \texttt{steffi.srusun.id}. Tujuan utamanya adalah agar aplikasi dapat langsung diakses secara luas berbasis web (tidak hanya berjalan secara lokal), sehingga mempermudah implementasi lapangan oleh pengguna. Seluruh rangkaian langkah \textit{deployment} pada fase ini dieksekusi melalui aplikasi \textit{terminal}, yang meliputi tahapan sebagai berikut.

\begin{enumerate}[label=\alph*.]
    \item Membuka aplikasi \textit{terminal} dan melakukan koneksi jarak jauh ke server menggunakan protokol SSH (\textit{Secure Shell}) melalui perintah \texttt{ssh -p 1996 srusun-steffi@steffi.srusun.id}, kemudian memasukkan kata sandi (\textit{password}) autentikasi server.

    \item Melakukan pengunggahan kode sumber (\textit{source code}) dari direktori lokal (berlokasi di \texttt{D:\textbackslash Kuliah\textbackslash TA\textbackslash TA1-v2\textbackslash Data Capture-3}) ke server menggunakan perintah \textit{Secure Copy Protocol} (\textit{SCP}). Perintah yang digunakan adalah \texttt{scp -P 1996 -R ./DataCapture6 srusun-steffi@steffi.srusun.id:\textasciitilde/htdocs/steffi.srusun.id/DataCapture6} untuk menyalin seluruh isi folder proyek ke direktori tujuan di server. Kemudian masukkan kembali kata sandi server untuk memvalidasi proses hingga seluruh berkas tersalin sepenuhnya.
    
    \item Menjalankan aplikasi pada server menggunakan \textit{process manager} \textbf{PM2} dengan perintah \texttt{pm2 start app.js}. Penggunaan PM2 bertujuan untuk memastikan aplikasi tetap berjalan dan dapat merespons permintaan melalui domain \texttt{steffi.srusun.id} secara terus-menerus.

    \item Apabila terdapat kebutuhan untuk melakukan perbaikan (\textit{bug fixing}) atau modifikasi langsung pada kode sumber tanpa harus mengunggah ulang, dapat menjalankan perintah \texttt{cd htdocs/steffi.srusun.id/DataCapture6} pada terminal. Selanjutnya, berkas dibuka dan diedit menggunakan perintah \textbf{nano} (contoh perintah: \texttt{nano public/index.html} untuk mengubah kode pada berkas \textit{index.html} di direktori \textit{public}). Setelah selesai edit berkas tertentu, berkas disimpan agar perubahan tersimpan pada sistem. 

    Agar perubahan kode tersebut dapat diterapkan sepenuhnya pada situs web, perlu dilakukan \textit{restart} layanan pada server dengan \textit{process manager} \textbf{PM2}. Langkah-langkahnya dimulai dengan menjalankan perintah \texttt{pm2 list} untuk melihat daftar serta status layanan yang sedang berjalan. Setelah mengidentifikasi identitas layanan yang dimaksud, jalankan perintah \texttt{pm2 restart [ID]} (contoh: \texttt{pm2 restart 0} apabila layanan aplikasi memiliki ID 0). Proses ini diakhiri dengan perintah \texttt{pm2 save} untuk memastikan konfigurasi status layanan tersebut tersimpan secara permanen pada server.
  
    \item Untuk inisialisasi awal dan pengelolaan basis data relasional dari \textit{terminal} server, gunakan perintah \texttt{mysql -u adrianjason -p}. Setelah melakukan autentikasi dengan memasukkan kata sandi basis data, syntax SQL dapat dieksekusi secara manual guna melakukan operasional CRUD (\textit{Create, Read, Update, Delete}). Beberapa perintah esensial yang digunakan meliputi:
    \begin{itemize}
        \item \texttt{CREATE DATABASE [nama\_db];} untuk membuat skema basis data baru di dalam server.
        \item \texttt{USE [nama\_db];} untuk memilih dan mengaktifkan basis data yang akan dikelola.
        \item \texttt{SHOW TABLES;} dan \texttt{DESCRIBE [nama\_tabel];} untuk meninjau daftar tabel serta spesifikasi kolom yang telah didefinisikan.
        \item \texttt{CREATE TABLE ...} untuk mendefinisikan struktur tabel baru beserta tipe datanya.
        \item \texttt{SELECT * FROM [nama\_tabel];} untuk menampilkan atau membaca (\textit{read}) seluruh data yang tersimpan di dalam tabel tertentu.
        \item \texttt{INSERT INTO [nama\_tabel] (...) VALUES (...);} untuk memasukkan (\textit{create}) data baru.
        \item \texttt{UPDATE [nama\_tabel] SET [kolom]=[nilai] WHERE [kondisi];} untuk memperbarui (\textit{update}) informasi tertentu pada baris data yang sudah ada.
        \item \texttt{DELETE FROM [nama\_tabel] WHERE [kondisi];} untuk menghapus (\textit{delete}) data tertentu (saat proses testing).
        \item \texttt{ALTER TABLE [nama\_tabel] ...} untuk memodifikasi struktur tabel yang sudah ada, seperti menambah atau menghapus kolom tanpa menghilangkan data yang telah tersimpan.
        \item \texttt{DROP TABLE [nama\_tabel];} untuk menghapus tabel beserta seluruh isinya secara permanen dari basis data.
    \end{itemize}


    
\end{enumerate}

\subsection{Fitur dan Implementasi Sistem}
Sistem dibangun menggunakan \textbf{Node.js}. Proses pengembangannya dilakukan secara iteratif yang dibagi ke dalam tiga tahapan transformasi antarmuka. Hal ini dilakukan untuk menyelaraskan fungsi sistem dengan kebutuhan riil di lapangan serta memastikan kesiapan integrasi dengan ekosistem aplikasi {Member} The Jarrdin.



\subsubsection{Evolusi dan Transformasi Antarmuka Sistem}
Transformasi antarmuka pengguna (UI) sistem mencerminkan kematangan sistem melalui proses iterasi berbasis Node.js sebagai berikut.

\begin{enumerate}[label=\Alph*.]
    \item \textbf{Tahap Awal (Sebelum Wawancara):} modul mandiri untuk validasi fungsi dasar. Fokus pada formulir {input} dan tabel data penyewa. Halaman Lihat Data hanya memiliki fitur input waktu \textit{check-out} dan \textit{checklist} komentar.
    \item \textbf{Tahap Pasca-Wawancara Pertama:} penambahan kolom NIB agen, perluasan ceklis komentar khusus Agen (visibilitas eksklusif \textit{Master Key}), dan fitur unduh \textit{log history}.
    \item \textbf{Tahap Pasca-Wawancara Kedua:} penerapan peran \textit{Master Key} (Ketua Agen, P3SRS, PKJ) dengan akses penuh, berbeda dengan Agen yang \textit{read-only} pada data sendiri. Penambahan Manajemen Unit, opsi Jenis Sewa di formulir, edit Jenis Sewa/Metode Pembayaran, filter pelanggaran, cek duplikat, dan {pop-up} detail log yang dapat diunduh.
    \item \textbf{Tahap Final: Integrasi Ekosistem Member:} sinkronisasi penuh. Penerapan SSO OTP, penyesuaian visual UI selaras aplikasi induk, dan penambahan menu akses langsung ``Penyewaan Unit'' pada beranda menu {Member}.
\end{enumerate}
